표시 방식 전환 (2026-07-23 확정)

유형별 카드 UI(인박스·게시글·명부)는 폐기하고, 모든 데이터 뷰를 엑셀형 시트로 통일한다 — 격자·행 번호·고정 헤더, 컬럼명은 실제 필드명(영문) 그대로. 이 탭은 진짜 DB를 보는 열람 전용이다: 승인·답글 같은 처리 버튼과 도착 배너·알림은 두지 않는다(그건 관리 화면·채팅의 몫). 남는 것 = 요약 대시보드 · 검색 · 필터 칩 · 엑셀 내보내기. 데이터 유형 메뉴는 사이드바에서 상단 탭으로 이동. 아래 템플릿 계약은 화면 생성이 아니라 시트의 컬럼 구성·기본 필터 축을 결정하는 용도로 유지된다.

핵심 원칙 — 화면은 AI가 그리지 않는다

플랫폼이 고정 템플릿 4유형을 제공하고, AI는 스키마를 만들 때 테이블마다 "템플릿 선택 + 슬롯 매핑" 메타데이터만 출력한다 — 자유 생성 금지, 골라서 채우기만 허용. 사용자에게는 개발 용어(테이블·행·컬럼) 대신 콘텐츠 이름(예약·회원·후기)으로, "조회"가 아니라 "처리할 일" 중심으로 보여준다.

모든 뷰 공통 (AI가 지정하지 않아도 슬롯에서 유도)
  • 검색(who/name/body 대상) · 필터 칩(status/replied/대표 속성 축, 건수 표기) · 엑셀 내보내기 · 페이지네이션
  • 상태 색과 행 액션 버튼([확정] 등)은 계약의 status.map/transitions가 결정 — 액션은 호버 숨김 없이 상시 노출
  • feed는 대량(수백~수천 건) 기본 내장: 요약 헤더(평균 별점·분포) · 미답변/저평점 필터 · 본문 2~3줄 클램프 · parent 소속 칩(발주→거래처)

도움말 장치 (2026-07-23 확정)

"DB가 뭔지 모르는 사장님"은 플랫폼 고정 카피로 해결한다 — AI가 판단·작성할 것을 늘리지 않는다.

  • 첫 진입 배너 — "여기는 자동으로 채워지는 엑셀이에요" 은유 + [관리 화면 가기]. 계정당 1회 노출, 닫으면 끝. 비유는 장부 대신 엑셀(세대 불문 + 화면 모양과 일치), 주어는 앱·홈페이지 어느 결과물이든 통하게 "사이트"로 중립화
  • 탭 우측 ? 버튼 — 누르면 같은 배너를 다시 연다 (상시 재열람 — "닫으면 끝"의 보완)
  • 관리 화면 이동 — 처리는 여기서 못 하므로, 배너 버튼·각 뷰 안내 문장의 "관리 화면" 링크로 관리·설정(운영 현황)에 바로 넘어간다 (settings.html?view=dash 딥링크)
  • 범위에서 뺀 것 — 표별 용어 설명(필드명 툴팁 등)은 앱마다 AI가 문장을 지어야 해 생성 하네스가 늘어남 → 초기 범위 제외. 문구는 전부 플랫폼 공통(어느 앱이든 동일)
