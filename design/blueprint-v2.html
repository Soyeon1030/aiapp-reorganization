핵심 원칙

이 화면은 기획 에이전트가 출력하는 JSON 하나를 그대로 그린다 — 화면 구조가 곧 계약이다. 에이전트 출력은 ① 사용자에게 "이렇게 이해했어요"를 보여주고 ② 구현 에이전트가 그대로 읽는다. 두 목적을 한 데이터로 해결한다.

섹션 ↔ JSON 매핑
화면 섹션JSON 키표현 규칙
제목·소개·칩metaneeds_auth → "회원 가입·로그인이 있는 서비스" 칩. routing → 초록(바로 생성)/주황(상담 필요) 칩
상담 배너meta.routing_reasonenterprise일 때만. "불가능" 금지 — 이유 + 상담 대안으로 서술
이렇게 이해했어요understanding[]confidence가 시각 언어의 핵심: confirmed = "말씀하신 내용", assumed = 주황 카드 + "추측했어요" — 사용자가 가장 먼저 확인할 지점
화면 흐름screens[]step 순서 가로 배치, is_core = 파란 테두리 + 핵심 태그. 3~5개(최대 6개)
저장하는 정보data_model[]엔티티 카드: label("예약 한 건마다 저장돼요") + 항목별 예시값 필수. 상태(선택)는 라이프사이클 필을 그대로 — 이 순서가 관리 화면 버튼이 됨
기본 화면 구성layouthome/dashboard/admin → 첫 화면·현황판·관리 화면 ("관리자" 단어 금지)
기능 구성features_mapreuse_baas = 초록 ✓(안심 신호), custom = 주황 ⚙(난이도 신호). custom 비면 "전부 준비된 기능" 문구
기획 문서로 받기documentsPRD·구조 설명서 다운로드. export_requires_credit → "크레딧이 사용돼요" (예상 수치 표기 금지)
카피 규칙
  • 개발 용어 금지: 테이블·필드·스키마 → "저장하는 정보"·"항목", 관리자 → "관리 화면"
  • 타입은 쉬운 말 7종 고정: 글자 / 숫자 / 날짜 / 전화 / 선택 / 이미지 / 참/거짓
  • 추측 항목을 조용히 두지 않는다 — 확정 바 힌트("주황색 추측 항목을 확인했다면")로 확인을 유도
  • 확정 문법은 blueprint와 동일: 클릭 즉시 진행 없음, [수정할래요]로 대화 복귀
미정 — 협의 필요
  • 이 화면의 위치: 설계안 대화 안의 카드 확장인지, 대화에서 여는 별도 전체 보기(현재 시안)인지
  • 추측 항목 개별 수정 UI(카드에서 바로 빼기/바꾸기) vs 채팅으로만 수정(현재 시안)
  • difficulty_level 숫자를 사용자에게 노출할지 — 현재 시안은 숨기고 라우팅 결과만 표시 (추천)
